# 🌡️ System Temperature & Thermal Monitoring

## Overview
In the context of the **Broker System**, "Temperature" is a concept of dual-purpose metric. It refers both to the **Hardware Health** (CPU/Server thermals) and **Market Kinetics** (the "heat" generated by trading volume and volatility).

This module ensures that as trading activity increases, the system remains within safe "thermal" operational bounds, preventing both hardware throttling and financial "meltdown."

---

## 1. The Thermal Analogy

| Financial Metric | Thermal Equivalent | Description |
| :--- | :--- | :--- |
| **Order Throughput** | **Kinetic Energy** | High frequency of `place_order()` calls increases the "vibration" of the system. |
| **Slippage/Fees** | **Friction** | Energy lost during the `execute_order()` phase that doesn't reach the final `Portfolio`. |
| **Risk Limits** | **Boiling Point** | The threshold where `RiskSnapshot` triggers a rejection to prevent system instability. |
| **Market Volatility** | **Ambient Temperature** | The external environment that makes it harder for the system to "cool down." |

## 2. API Integration    

### `SystemHealth` Monitoring
To track actual physical temperature, we extend the `common_types.h` logic

```cpp
struct ThermalState {
    double cpu_temp_celsius;
    double fan_speed_rpm;
    bool is_throttling;
};

// Returns a Status if the hardware is overheating
Status check_thermal_safety(const ThermalState& state);
```

### Thermal Risk Validation
The `risk_engine.h` can be conceptualized as a cooling system. If the `gross_exposure_cents` rises too quickly, the "Temperature" of the account rises.

> **System Invariant:** If `ThermalState.cpu_temp_celsius > 85.0`, the `OrderService` must automatically switch to `REJECT_ALL` mode to protect hardware integrity.


## 3. Operational States

### ❄️ Cold (Idle)
* **Activity**: Low `OrderRequest` volume.
* **Risk**: Minimal `gross_exposure_cents`.
* **Action**: System in power-save or low-latency waiting mode.

### ☀️ Warm (Active)
* **Activity**: Consistent `Fill` events in `execution_service.h`.
* **Risk**: Managed exposure within `Portfolio` limits.
* **Action**: Optimal operating state; high efficiency.

### 🔥 Hot (Volatile)
* **Activity**: Rapid-fire `cancel_order` and `place_order` calls.
* **Risk**: Approaching `max_order_notional_cents`.
* **Action**: The Risk Engine increases "cooling" by tightening validation logic.



## 4. Visualizing System Entropy


As the system moves toward higher "heat," the probability of a `Status::failure` increases. Monitoring these curves allows the Broker System to predict failures before they happen.


## 5. Maintenance
* **Hardware**: Ensure server racks have adequate airflow for high-frequency trading.
* **Software**: Optimize `apply_execution` to reduce the "computational friction" (CPU cycles) per trade.
